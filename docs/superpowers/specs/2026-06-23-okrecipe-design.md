# OKRecipe 設計書

- 作成日: 2026-06-23
- リポジトリ: https://github.com/nitda-lab/OKRecipe

## 1. 概要 / 目的

冷蔵庫の在庫を管理し、それを使い切るレシピを提案するWebサービス。
在庫から作れるレシピだけでなく、「何を買って何を作れば食材が余らないか」まで踏み込んで提案する。
在庫入力は手入力を極力避け、**写真（冷蔵庫の中身 / レシート）** で行う。

### ターゲット
- まずは2人で利用。将来的に一般公開も視野に入れ、最初から認証・コスト構造・拡張性を意識した作りにする。

### 設計の中心思想
- **AIとの対話が主役の操作インターフェース**。在庫・保存済みレシピなどは、AIなしでも完結して動く**独立システム（信頼できる土台）**として持つ。
- AIはその独立システムを**読み書きする操作役（tool calling）**であり、データの「真実」は常に独立システム（DB）が持つ。
- 操作経路は「写真取り込み」「手動画面」「AI対話」の3つ。経路は3つでも、更新先のデータは1つ。

## 2. 全体構成

```
[スマホブラウザ（モバイルファースト / PCもレスポンシブ）]
   │
   ▼
[Next.js (Vercel)]
   ├─ UI (React)
   ├─ API Route: /api/ingest    … 写真 → 食材抽出
   ├─ API Route: /api/inventory … 在庫CRUD・補正
   ├─ API Route: /api/recipes   … レシピ保存・一覧
   └─ API Route: /api/chat      … AI対話（tool calling）
   │
   ├──▶ [Supabase] 認証 / Postgres(在庫・レシピ) / Storage(写真)
   └──▶ [AIプロバイダ層] OpenAI互換I/F → nanoGPT
```

### 技術スタック（案A: Next.js + Supabase 一体型）
- **フロント/バック**: Next.js（React）。フロントとAPIを一つの枠で。
- **DB/認証/ストレージ**: Supabase（Postgres / Auth / Storage）。2人運用はほぼ無料、公開時もそのまま伸ばせる。
- **デプロイ**: Vercel。
- 理由: 「ユーザー認証」「写真ファイルの保存」「DB」という地味に面倒な部分をSupabaseが一気に片付け、核心（在庫ロジック・AI連携）に集中できる。Reactの学習がそのまま実戦になる。

### AIプロバイダ層
- AIへのアクセスは**薄い1モジュール**に閉じ込める。外には `chat(messages, model, tools)` 相当の関数だけ見せ、中身（nanoGPT / OpenAI 等）を隠す。
- **OpenAI互換I/Fに統一**。運用はnanoGPTで開始し、必要なら差し替え可能。
- **モデルは用途別に指定**できる（プロバイダ層で吸収）:
  - 画像認識（在庫抽出）用: vision対応モデル（GPT-4o / Gemini / Claude 等）
  - 対話（tool calling）用: function calling対応モデル（GPT-4o系 等）
- nanoGPTのチャット補完APIはOpenAI互換で画像入力（`image_url`、base64 or HTTPS URL、PNG/JPEG/JPG/WebP）に対応済みであることを確認済み。

> 補足: Claude Pro（消費者向けサブスク）はバックエンドに組み込めない（APIキーが付かない）。バックエンドからのAI利用にはAnthropic APIかnanoGPT等のAPIが必要。本サービスはnanoGPTを主軸とする。

## 3. データモデル（Postgres / Supabase）

```
users              … Supabase Auth（2人 → 公開対応）

ingredients        … 食材マスタ（正規化用）
  id, name, category, default_unit

inventory_items    … 在庫
  id, user_id, ingredient_id,
  quantity_text   … 自然言語の個数そのまま ("2個","一人前分","残り半分")
  quantity_num    … 可能なら補助的な数値（"2個"→2。無ければnull）
  unit_text       … "個","人前","回分" 等（補助。無くても可）
  expires_at      … 任意
  source          … receipt / fridge_photo / manual / chat
  updated_at

ingest_logs        … 写真取り込みの記録（監査・やり直し用）
  id, user_id, type(receipt/fridge), image_path,
  ai_raw_json, status, created_at

recipes            … 提案 / 保存・作ったレシピ
  id, user_id, title, ingredients_json(必要量), steps_json,
  status(suggested/saved/cooked), cooked_at

shopping_items     … 買い物提案（献立から逆算）
  id, user_id, ingredient_id, quantity_text, plan_id
```

### 単位の扱い（自然言語）
- 個数は「一個・一人前・一回分・残り半分」のような自然言語を許容する。数値での厳密計算を前提にしない。
- そのため在庫の減算も「厳密な引き算」ではなく、**AIが文脈で解釈して更新**する方式（4章参照）。
- `quantity_num` / `unit_text` は可能な範囲で補助的に持つ（並び替えや期限警告などに利用）。

## 4. 独立システムとAI対話レイヤー

```
[独立システム＝信頼できる土台（AIなしでも完結して動く）]
  ・在庫システム       … DB＋CRUD。手動画面から直接操作可
  ・レシピ保存システム  … 保存済みレシピのCRUD・一覧
  ・写真取り込みシステム … 画像 → 食材抽出 → 在庫反映

        ▲ 道具(tools)として呼ぶだけ
        │
[AI対話レイヤー＝上に乗る操作インターフェース]
  ・チャットで独立システムを呼び出す
  ・クイックボタン = 定型プロンプトを投げる入口
```

### AIが呼べる道具（tools / function calling）
```
list_inventory()                       … 今の在庫を読む
add_inventory(name, qty_text)          … 「卵買った」→ 追加
update_inventory(id, qty_text)         … 「卵2個使った」→ 減らす/変更
remove_inventory(id)                   … 使い切り
suggest_recipe(条件)                    … 単発レシピ提案
create_meal_plan(日数, 余らせたい食材)   … 週次献立＋買い物リスト生成
save_recipe(recipe)                    … レシピを保存
```

### 対話例
- 「冷蔵庫の余りもので何か作れる?」→ AIが `list_inventory` → 提案
- 「それ作ったよ、卵2個と牛乳使った」→ AIが `update_inventory` を呼んで在庫を自動で減らす（実行前に確認）

### クイックボタン（よく使う対話の1タップ入口）
- 「ありものでできるレシピ」「今週の献立を組んで」「賞味期限が近いものを優先」など。
- 中身は決め打ちのプロンプトをチャットに送るだけ。

### レシピ提案の単位（ハイブリッド）
- 普段は在庫から作れる**単発レシピ**を提案。
- 「献立を組んで」と頼まれたら数日〜1週間の**献立プラン＋買い物リスト**を生成（食材を使い切る計画）。

## 5. 主要フロー

### 5.1 写真取り込みフロー
```
[写真を撮る/選ぶ]（レシート or 冷蔵庫）
      ▼
[AIが食材を抽出]
      ▼
[取り込み確認リスト（下書き）]
   ・検出された食材が一覧で並ぶ
   ・各行: 名前 / 個数(自然言語) を編集可
   ・行の追加（検出漏れを手動で足す）
   ・行の削除（誤検出を消す）
   ・冷蔵庫写真なら「現状で上書き」、レシートなら「在庫に加算」を選択
      ▼ [確定]
[在庫一覧に反映] ← ここから通常の在庫システム（いつでも再編集可）
```
- 確定するまで在庫に入らない（誤認識が在庫を汚さない）。
- 元画像はStorageへ、生の抽出結果は `ingest_logs` に保存（やり直し・追跡用）。

### 5.2 在庫の更新経路（3つ、更新先は1つ）
1. **写真取り込み**: 上記フローで反映（レシート=加算 / 冷蔵庫写真=現状補正）
2. **手動画面**: 在庫一覧から追加・個数変更・削除のCRUD
3. **AI対話**: tool callingで `add/update/remove_inventory`

### 5.3 「作った」→ 減算
- レシピを「作った」にする、または対話で「これ作った」と伝えると、材料を在庫から引く。
- 自然言語単位のため厳密計算せず、AIが解釈し**確認を挟んで**更新。

## 6. エラー処理・整合性
- **AIの在庫更新は確認を挟む**: いきなり消さず「卵を2個減らして在庫1個にします。OK?」と確認 → 暴走防止。
- **写真取り込みは下書き経由**: 抽出結果をそのまま反映せず、確認画面で修正 → 確定。
- **AI障害時**: チャットが使えなくても、在庫・レシピの手動画面は通常通り動く（土台が独立しているため）。
- **取り込みログ保持**: `ingest_logs` に生の抽出結果を残し、やり直し・原因追跡が可能。

## 7. テスト方針
- **独立システム（在庫/レシピCRUD）**: 純粋なロジックとしてユニットテスト（AI不要で完結）。最重要。
- **AIプロバイダ層**: モック応答で tool calling の流れを検証（実APIは叩かない）。
- **写真取り込み**: 抽出結果JSON → 在庫反映の変換ロジックをテスト（画像認識自体は固定サンプルで確認）。
- **手動E2E**: 主要フロー（写真→確認→在庫 / 対話→提案→作った→減算）をスマホ実機で目視確認。

## 8. 確定した設計判断（まとめ）
- ターゲット: まず2人、将来公開視野。認証・コスト・拡張性を最初から意識。
- 入力: 冷蔵庫写真とレシート写真の両方を最初から。
- 在庫の減り方: レシピ「作った」で自動減算 + たまに写真で現状補正、の併用。
- レシピ提案: 単発提案 + 依頼時に週次献立（ハイブリッド）。
- スタック: Next.js + Supabase（案A）、モバイルファースト。
- AI: OpenAI互換I/Fで抽象化、nanoGPTで開始。画像認識・対話とも可。
- 単位: 自然言語（一個・一人前・一回分 等）。厳密計算しない。
- 操作: AI対話が主役。在庫・レシピは独立システム（手動画面・クイックボタンは補助）。
- 写真取り込み: 撮る → 確認リストで編集 → 確定 → 在庫へ。
