# 冷蔵庫写真の取りこぼし改善（画像認識の精度）設計

- 日付: 2026-06-24
- 対象: 写真取り込み（冷蔵庫）の食材抽出
- スコープ: Approach A（プロンプト改訂 + `max_tokens` 種別別）のみ

## 背景・課題

冷蔵庫写真から食材を抽出する際、**写っているのに抽出されない「取りこぼし」**が起きる。
原因を調査した結果、主因は次の2点（いずれも `src/lib/ai/vision.ts`）。

1. **プロンプトが間引きを指示している（最大の自爆点）**
   冷蔵庫プロンプトに「調味料の小瓶など細かすぎるものは主要なものに絞る」とあり、
   モデルに列挙せず減らすよう明示的に命じている。
2. **`max_tokens` が 800**
   品数の多い冷蔵庫では JSON 配列が途中で切れ、後半が丸ごと欠落しうる。

なお画像は `PhotoIngest.tsx` で**縮小せず原寸**送信されており、画質劣化による取りこぼしではない。

### 本設計に含めないもの（別件）

- **撮影経路の変化（`capture="environment"` 削除）**: 複数画像対応のコミット（ef41594）で、
  入力が「背面カメラ直接撮影」から「ギャラリー/ファイル選択」に変わっている。
  ギャラリー経由で再圧縮・縮小された画像が渡され精度が落ちる可能性があるが、
  本設計のスコープ外（効果を見てから別途検討）。
- レシートプロンプトの変更（今回の症状は冷蔵庫のため触らない）。
- vision モデルの変更（Approach B。本設計で足りなければ別途）。

## 変更対象

`src/lib/ai/vision.ts` のみ。API Route・UI は変更しない。

## 設計

### 1. 冷蔵庫プロンプトの全面改訂

「主要なものに絞る」を撤廃し、**網羅列挙**へ反転する。要点:

- 棚 → ドアポケット → 引き出し の順に走査し、見えるものは全部挙げる。
- 一部しか見えない／包装で隠れていても、判別できれば挙げる。
- 不確実でも挙げてよい（`qty_text` は不明なら `"あり"`）。
- 調味料・小瓶も含める。間引かない。
- 食品/飲料/食材のみ（容器・家電・宣伝文は除外）。
- これまで通り **JSON配列だけ** を返す（形式: `[{"name":"卵","qty_text":"2個"}]`）。

出力形式は変えないため、`parseExtraction` は無変更で互換。

### 2. `max_tokens` を種別ごとに

品数の多い冷蔵庫で JSON が途中で切れないよう、種別ごとにトークン上限を持つ。

- `receipt`: 1000
- `fridge`: 2500
- `deps.maxTokens` が明示的に渡された場合はそれを優先（上書き可）。

`createVisionExtractor` に「種別 → トークン数」のマップを持たせ、`extract(_, kind)` で選択する。

### 3. テスト（TDD）

- 既存テストは維持する（`parseExtraction`、`extract` の基本動作）。
- 追加: `extract(..., 'fridge')` のリクエスト body の `max_tokens` が冷蔵庫向けの値（2500）に
  なっていることを検証する。`extract(..., 'receipt')` は 1000。
  → 受信側（route/UI）を変えずに送信パラメータを担保する。
- プロンプト文面の品質は単体テスト不能。**手元の冷蔵庫写真での目視確認**で検証する。

## 検証方法

改訂前後で同じ冷蔵庫写真を投げ、取りこぼしが減るかを目視比較する。
不足する場合は Approach B（モデル変更）または撮影経路の別件対応へ進む。

## リスク

- プロンプトを網羅寄りにすることで、**過剰抽出（食品でないものや重複）が増える**可能性がある。
  ただし取り込みは確認・編集画面（`PhotoIngest`）でユーザーが削除できるため、
  「取りこぼし」よりは許容しやすい（消す方が足すより楽）。
- `max_tokens` 引き上げによるコスト・レイテンシ増は軽微（必要な分だけ生成される）。
