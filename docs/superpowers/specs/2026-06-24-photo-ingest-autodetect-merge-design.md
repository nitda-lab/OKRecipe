# 写真取り込み：自動判別＋複数枚統合 設計

- 日付: 2026-06-24
- 対象: 写真取り込み（レシート/冷蔵庫）の食材抽出フロー

## 背景・課題

1. **手動でレシート/冷蔵庫を選ぶのが面倒** → 自動判別したい。
2. **複数枚添付すると同じ食材が重複して認識される** → 統合したい。
   複数枚は「同一冷蔵庫の別アングル」とは限らず、「冷蔵庫の中＋机に並べた材料」のように
   **別シーンを合わせて手持ちの食材ぜんぶを1つの在庫にしたい**ケースもある。

「同じ物が2枚に写った」のか「本当に2個ある」のかは、抽出後に名前だけで突き合わせても
原理的に区別できない。よって後段のクライアント側 dedup ではなく、**全画像を1回のvision呼び出しに
まとめて渡し、モデルが視覚的に統合・重複排除する**方式を採る。実測で現行 `gemma-4-31b-it` が
1メッセージ複数画像を受け取り、机＋冷蔵庫の統合・重複排除（同一MILKを1つに）できることを確認済み。

## アーキテクチャ（データフロー）

複数枚を **1回のvision呼び出し** にまとめ、モデルが「判別 → 各画像から抽出 → 統合・重複排除」を
一括で行い、**統合済みリスト**を返す。クライアント側の名前マッチング dedup は廃止（不要）。

```
PhotoIngest（全画像をダウンスケール → 1回POST）
  → /api/ingest（images配列を受け、extract(images)）
    → vision.extract(images)（統合プロンプト＋複数image_url、1回呼び出し）
      → { kind, items }（統合済み）
  ← items を下書きへ。ユーザーが確認・編集して在庫へ
```

## 各コンポーネントの設計

### 1. `src/lib/ai/vision.ts`

- **signature変更**: `extract(imageDataUrl: string, kind)` → `extract(images: string[]): Promise<{ kind: 'receipt' | 'fridge'; items: ExtractedItem[] }>`。kind引数は廃止。
- **統合プロンプト**（2つの `PROMPTS` を1つに集約）。要点:
  - これは「今手持ちの食材」を写した1枚以上の写真。レシート・冷蔵庫の中・机に並べた材料が混在しうる。
  - 全写真を合わせて手持ちの食材を **1つのリストに統合** する。
  - レシートは購入食品のみ抽出（税・合計・袋・店名など非食品は除外）。
  - 冷蔵庫/並べた食材は、棚→ドアポケット→引き出しの順に走査し、隠れていても判別できれば挙げる（漏れなく）。
  - 複数写真に同じ物が写っていたら **1つにまとめ重複させない**。ただし明らかに別個体が複数あるならその数を反映。
  - 個数は自然言語、不明なら `"あり"`。英語/ローマ字は日本語に直す。
  - 出力は次の形式のJSONのみ: `{"kind":"fridge","items":[{"name":"卵","qty_text":"6個"}]}`。
    kind はレシートが主なら `"receipt"`、それ以外は `"fridge"`。
- **`max_tokens`**: 種別マップを廃し単一値 **3000**（複数枚・大量列挙に備える）。`deps.maxTokens` で上書き可。
- **パース**:
  - 既存 `parseExtraction(text): ExtractedItem[]` は維持しつつ、**切り詰め耐性**を追加する。
    `max_tokens` で配列が途中で切れて全体が不正JSONになっても、**完結している要素だけを救出**して返す
    （現状は不正JSONで `[]`＝全損になる）。実装: 全体 `JSON.parse` に失敗したら、
    `}` 区切りで要素単位に走査し、`JSON.parse` できた `{name, qty_text}` だけを採用する。
  - kind を読む薄いラッパ `parseResult(text): { kind, items }` を追加。items は `parseExtraction` に委譲。
    kind はテキスト中の `"kind"` 値を簡易判定し、`"receipt"` を検出したときのみ `'receipt'`、
    それ以外（素の配列を含む）は `'fridge'` にフォールバック（出力ブレに頑健）。

### 2. `src/app/api/ingest/route.ts`

- 入力を `image + kind` → **`images: string[]`**（非空・各要素は data URL 文字列）に変更。
- `extract(images)` を呼び、`{ kind, items }` の kind で従来通り `ingest_logs` に記録、`{ items }` を返す。
- **DBスキーマ変更なし**（kind は引き続き `'receipt' | 'fridge'` のどちらか）。
- バリデーション: `images` が非空配列でなければ 400。vision失敗は 502（現行踏襲）。ログ失敗は握りつぶす（現行踏襲）。

### 3. `src/components/PhotoIngest.tsx`

- **レシート/冷蔵庫トグルを撤去**（自動判別）。`kind` state を削除。
- **画像ダウンスケール（必須）**: 送信前に全画像を canvas で長辺最大 **1568px** に縮小し **JPEG 品質0.8** で
  data URL 化する。これにより複数枚を1リクエストに束ねても **Vercelのボディ上限（約4.5MB）** を超えない。
  純関数として切り出してテスト可能にする（`src/lib/image/downscale.ts` に `downscaleDataUrl` 等）。
  ただし canvas はブラウザAPIのため、純粋部分（長辺から縮小後の幅・高さを算出する `fitWithin`）を
  ユニットテスト対象とし、canvas描画自体は手動検証とする。
- **送信を1回に**: `send()` を「1枚ずつN回POST＋concat」→ **全枚をダウンスケールして配列で1回POST** に変更。
  レスポンスの統合済み items をそのまま下書きへ反映（クライアント側 concat / dedup は廃止）。
- **最大枚数**: 一度に **8枚** まで。超過時はUIで弾き、メッセージ表示。
- add/overwrite トグルは維持。

## 制約・トレードオフ（仕様として明記）

- **1回呼び出しの全か無か**: 従来は1枚失敗でも他を収集できたが、統合のため1回呼び出しにするので
  失敗すると全滅する。許容する。
- **kind は近似**: 混在バッチの kind はモデルの推測で、用途は監査ログのみ。
- **同一物 vs 別個体の判定はモデル任せ**: 完全ではない。最終的にユーザーが下書きで確認・編集する前提。
- **ダウンスケールは画質とサイズのトレードオフ**: 長辺1568px/JPEG0.8 は vision 精度に十分な範囲で
  ボディ上限回避を優先した既定値。

## テスト（TDD）

- `parseExtraction`: 既存テスト維持。追加で **切り詰め救出**（途中で切れた配列から完結要素だけ返す）を検証。
- `parseResult`: `{kind, items}` を読む／素の配列なら kind=`fridge` フォールバック／`receipt` 検出。
- `extract(images)`: **N枚を1メッセージにN個の image_url として送る**こと、単一 `max_tokens`(3000) を送ること、
  戻り値が `{kind, items}` であることを fetch モックで検証。
- `fitWithin`（ダウンスケールの寸法計算）: 長辺が上限以下ならそのまま／超過時は縦横比維持で縮小、を検証。
- UI（PhotoIngest）と canvas 描画はコンポーネントテストが無いため **手動検証**。

## 手動検証

- 1枚（レシート）/1枚（冷蔵庫）/複数枚（冷蔵庫の別アングル）/複数枚（冷蔵庫＋机）で `/ingest` を実行し、
  自動判別・統合・重複排除・ボディ上限内に収まることを実機で確認する。
